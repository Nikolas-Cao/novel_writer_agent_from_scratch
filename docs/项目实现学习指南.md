# 百万字分章小说 Agent — 项目实现学习指南

本文档帮助从零理解本项目的整体架构与实现，按「先概念、再结构、后细节」的顺序组织。

---

## 一、项目目标与核心能力

- **分章生成百万字级小说**：按章为粒度生成与存储，支持断点续写。
- **前文 + 大纲驱动**：RAG 检索前文摘要 + 人物图谱 + 当前章大纲，作为写章上下文。
- **用户反馈 → 重写 → 可选更新大纲**：对某章提交反馈后重写，并可选择反向更新大纲。
- **最新章安全编辑约束**：仅允许对“最新章节”执行重写/重生成，避免中间章改动导致后续章节断裂。
- **章节回滚**：可将项目回滚到任意已生成章节，自动删除该章之后的正文、RAG 摘要与人物图谱快照。
- **项目列表右键管理**：支持项目级重命名（可选昵称）与硬删除（删除项目私有资源）。
- **Web 前端**：新建小说、选剧情概要、看大纲、续写、反馈重写等。
- **新建小说流程**：输入意图 → LLM 生成多个剧情概要 → 选/自定义概要 → 生成全书大纲 → 开始写章。
- **单章 Markdown 输出**：每章以 `.md` 存储，支持标题/段落/强调及插图语法。
- **章节插图（可选）**：用 LLM 选出**单张**最佳插图位并生成详细画面描述，再调用 **OpenAI Images API** 生图；成功则插入 Markdown，**失败则跳过插图**（不用占位图）。图片落在 `projects/{project_id}/chapters/{章号}/images/`。
- **人物关系按章查看**：后端按章节返回人物图谱快照，前端支持弹窗查看当前章人物与关系。
- **全本地方案**：向量检索、状态持久化、Checkpoint 均使用本地存储，无云依赖。
- **同人知识库（可选）**：全局命名知识集（首期 `txt`/`md`），项目可勾选 `selected_kb_ids`；`plan_outline` / `write_chapter` / `refine_chapter` 注入分层摘要 + 向量/关键词混合检索；**二创改动（`canon_overrides`）优先于原著**；仅在大纲生成前允许 `PATCH` 修改绑定。

## 三、配置与状态（入门必读）

### 3.1 config.py

- **模型与 API**：从环境变量读取，支持 PLANNER / WRITER 分离（大纲/概要用 planner，写章/润色用 writer）。默认可与 Ollama 等本地服务对接。
- **写作参数**：`CHAPTER_WORD_TARGET`（每章目标字数）、`DEFAULT_TOTAL_CHAPTERS`、`PLOT_IDEAS_COUNT`、`RAG_PREVIOUS_CHAPTERS`（写章时取前几章摘要）、`CHARACTER_GRAPH_RECENT_CHAPTERS`（写章时人物关系摘要的章节滑窗宽度）。
- **本地路径**：`PROJECTS_ROOT`（项目目录）、`CHECKPOINT_DIR`（状态/checkpoint）、`VECTOR_STORE_DIR`（RAG 向量库）。默认 `VECTOR_STORE_DIR` 为仓库下独立目录 `vector_store/`（**不在** `PROJECTS_ROOT/{project_id}/` 内）；Chroma 数据在 `VECTOR_STORE_DIR/{project_id}/`。**全局知识库**数据在 `VECTOR_STORE_DIR/global_kb/`（`registry.json`、每 `kb_id` 子目录：原始文件、Chroma、SQLite FTS、构建任务 `jobs/`、分层摘要 `assets/`）。每个项目在正文与图谱侧对应 `PROJECTS_ROOT/{project_id}/`，其下有 `chapters/`（含 `chapters/{章号}/images/` 存插图）、可选历史 `images/`、`character_graph.json`、`character_graph/{chapter_index:03d}.json`。
- **DEBUG 调用落盘**：`DEBUG=true/1/yes/on` 时，每次文本 LLM 调用会写入 `PROJECTS_ROOT/{project_id}/llm_invoke_results/`；文件名格式为 `{utc_timestamp}_{status}_{purpose}_{short_id}.json`。其中 `status` 为 `success/error`，`purpose` 为细粒度业务目的（如 `outline_extend_window`、`write_chapter`、`refine_chapter`），并在 JSON 中保留 `purpose_group` 便于聚合。流式调用会把所有 chunk 聚合成完整 `response_text` 后统一写入一条记录。
- **知识库相关环境变量**（均在 `config.py`，可调分块/批处理/检索/工具循环上限等）：`KB_CHUNK_*`、`KB_INGEST_BATCH_CHUNKS`、`KB_READ_BLOCK_BYTES`、`KB_MAX_CHUNKS_PER_DOCUMENT`、`KB_ASSET_*`、`KB_RETRIEVE_*`、`KB_TOOL_LOOP_MAX_CALLS`。
- **插图生图**：`IMAGE_GEN_API_KEY`（默认回退 `OPENAI_API_KEY` / `PLANNER_API_KEY`）、`IMAGE_GEN_BASE_URL`（空则官方 OpenAI）、`IMAGE_GEN_MODEL`（默认 `dall-e-3`）、`IMAGE_GEN_SIZE`、`IMAGE_GEN_TIMEOUT_S`。生图成功后若响应含 `usage`（如 `gpt-image-1`），会累计到 `state.token_usage[IMAGE_GEN_MODEL]`，与 LLM 用量同一结构；仅 `total_tokens` 时记入 **output_tokens** 侧便于展示。

### 3.2 state.py — 核心数据结构

- **OutlineStructure**：`volumes[]` → 每卷 `chapters[]` → 每章 `title`、`points[]`、可选 `conflict`。有 `outline_structure_to_string()` 供 LLM 使用。
- **ChapterMeta**：章节元数据（chapter_id、title、summary、path_or_content_ref、word_count、index、可选 images_refs）。
- **CharacterGraph**：`nodes[]`（id、name、aliases、description、first_chapter）、`edges[]`（from_id、to_id、relation、note、可选 **first_chapter**：关系首次出现章，供写章时与滑窗配合过滤）。
- **NovelProjectState**：LangGraph 的共享状态，包含：
  - 创作意图与概要：instruction、plot_ideas、selected_plot_summary
  - 大纲：outline、outline_structure
  - 章节：chapters、current_chapter_index、current_chapter_draft、current_chapter_final
  - 人物：character_graph
  - 反馈与重写：user_feedback、last_rewrite_draft、update_outline_on_feedback
  - 配置：project_id、total_chapters、chapter_word_target、chapter_output_format、enable_chapter_illustrations
  - RAG/插图中间结果：retrieved_summaries、retrieved_outline_chunk、character_context_summary、last_chapter_summary、**illustration_point**、**illustration_prompt**、**illustration_asset**（单图）、以及兼容旧字段的 illustration_points、illustration_assets
  - 观测与生命周期（服务端常用）：**token_usage**（按模型汇总 input/output tokens，流式场景可与追踪包装配合）、**created_at**（Unix 秒，创建时间；空项目清理也会参考状态文件/目录 mtime）
  - **同人知识库**：**selected_kb_ids**、**kb_enabled**；**canon_overrides**（二创相对原著的差异清单，检索与润色时最高优先）；写章侧可观测字段如 **kb_assets_text** / **kb_evidence_text** / **kb_confidence**、**consistency_report**

理解 State 的字段与流转，是理解整条工作流的关键。

## 四、本地存储层

### 4.1 storage/chapter_store.py

- **ChapterStore(root)**：根目录默认 `PROJECTS_ROOT`。
- **path_for(project_id, chapter_index)** → `{project_id}/chapters/{index:03d}.md`
- **save(project_id, chapter_index, content)**：写入 .md，返回相对路径 ref。
- **load(project_id, chapter_index)** / **load_by_ref(ref)**：按路径读内容。

章节正文只存文件，元数据在 state 的 `chapters` 里。

### 4.2 storage/character_graph.py

- **CharacterGraphStore(root)**：同时维护：
  - 兼容聚合文件：`{project_id}/character_graph.json`
  - 按章快照：`{project_id}/character_graph/{chapter_index:03d}.json`
- **load_for_chapter(project_id, chapter_index)**：读取 `<= chapter_index` 的最近快照；若无快照回退到聚合文件。
- **merge(..., chapter_index=idx, base_graph=...)**：按 id 去重节点、按 `(from_id,to_id,relation)` 去重关系，并保存为该章快照；同时刷新聚合文件为最新结果。
- **delete_snapshots_from(project_id, start_index)** / **refresh_legacy_latest(project_id)**：给章节回滚使用，删除尾部快照并对齐 legacy 聚合图谱。

## 五、RAG（本地向量检索）

### 5.1 rag/embedding.py

- **LocalHashEmbeddingFunction**：基于 token 的轻量本地 embedding（无网络），满足 Chroma 接口，用于本机可复现的索引与检索。

### 5.2 rag/indexer.py

- **LocalRagIndexer(root)**：每个项目一个 Chroma 持久化目录（`root/{project_id}`），collection 名 `novel_rag`。
- **add_chapter_summary(project_id, chapter_index, summary_text)**：写入章节摘要，id 如 `summary-{chapter_index}`。
- **add_outline_chunk(project_id, chunk_text, volume_idx, chapter_idx)**：写入大纲片段（可选）。
- **upsert_outline_chunks_for_chapters(...)**：按章节索引批量回写大纲片段（反馈改纲后与 RAG 保持一致）。
- **delete_chapter_summaries_from(project_id, start_index)**：删除某章之后的摘要（章节回滚时使用）。
- **delete_project(project_id)**：删除该项目向量目录（测试/清理用）。

### 5.3 rag/retriever.py

- **LocalRagRetriever(root)**：同一套 Chroma 目录与 embedding。
- **retrieve_for_chapter(project_id, current_chapter_index, k_chapters, k_outline)**：
  - 取 `kind==chapter_summary` 且 `chapter_index < current` 的文档，按章号排序后取最近 `k_chapters` 条；
  - 取 `kind==outline_chunk` 且 `chapter_index == current` 的文档作为当前章大纲片段；
  - 返回 `{summaries: [{chapter_index, text}], outline_chunk: str}`。

写章节点用该结果拼进 prompt，实现「前文 + 当前章大纲」的上下文。

### 5.4 全局知识库 RAG（rag/global_kb_*.py + knowledge_base/）

- **存储**：`KnowledgeBaseStore(VECTOR_STORE_DIR)` 管理 `global_kb/registry.json`、每知识集元数据与文档列表、异步构建任务状态、按文档/合并后的分层摘要 JSON。
- **摄取**：`knowledge_base/ingest.py` 按字节块流式读入 `txt`/`md`，分块后批量写入 **GlobalKbChroma**（每 `kb_id` 独立持久化目录）与 **SQLite FTS5**（`knowledge_base/fts_index.py`，支持按文档替换索引）。
- **检索**：`GlobalKbRetriever` 先拼接已就绪的 **分层摘要资产**（`characters` / `timeline` / `world_rules` / `core_facts`），再对所选 `kb_ids` 做 Chroma 语义召回 + FTS 关键词召回，合并去重后供节点使用。
- **编排**：`graph/knowledge_context.py` 提供 `build_kb_context_for_outline`、`build_kb_context_for_writing` 等：默认多步查询；在证据不足等规则下可触发**受限**补查（次数与 token 预算由配置约束）。未绑定知识库时各节点走原有分支，行为与旧版一致。

## 六、记忆与状态持久化

### 6.1 memory/local_checkpointer.py

- **LocalFileCheckpointer(root)**：按 `thread_id` 存 JSON 文件（如 `checkpoints/api_state/{project_id}.json`），实现「按项目恢复状态」。
- **try_create_sqlite_checkpointer()**：若已安装 LangGraph 的 SqliteSaver，可返回 SqliteSaver 供 workflow 使用；否则返回 None，服务端用 LocalFileCheckpointer 存 state。

当前 **server.py** 使用的是 LocalFileCheckpointer，状态在每次 API 调用后显式 load/save，而不是通过 LangGraph 的 thread_id 自动恢复；这样更简单且与「项目即会话」一致。

## 七、工作流节点（graph/nodes）

每个节点都是 **async 函数**，接收 `state: NovelProjectState`，返回要写回 state 的 **Dict**（LangGraph 会做浅合并）。

### 7.1 generate_plot_ideas

- **输入**：state 中的 `instruction`。
- **行为**：用 planner_llm 生成 `PLOT_IDEAS_COUNT` 条剧情概要，解析为 `List[str]`（支持 JSON 或兜底解析）。
- **返回**：`{plot_ideas: List[str]}`。

### 7.2 plan_outline

- **输入**：`selected_plot_summary`、`total_chapters`；可选异步回调 `on_progress(stage, message)`（供服务端 NDJSON 流式上报进度）；可选 **`kb_context`** 字符串（服务端由 `build_kb_context_for_outline` 注入所选全局知识库摘要）。
- **行为**：
  - 当 `total_chapters` ≤ `PLAN_OUTLINE_SINGLE_CALL_MAX`（默认 16）时：走 **短篇单次生成**（`outline_short`），直接输出可写作大纲。
  - 当章节数更大时：走 **长篇滚动扩窗**（`outline_skeleton_lite -> outline_extend_window`）：
    1) 先按批循环生成全书极简骨架（默认 5 批；每章含 `global_index/title/description`，并做覆盖校验）；
    2) 首窗按 `OUTLINE_WINDOW_SIZE`（默认 10）生成详细章节；
    3) 后续在边界触发时继续按窗扩写。
  - 收口阶段统一执行 `outline_finalize`：生成 `outline` 字符串、可选写入 RAG、可选提取并合并 `canon_overrides`。
- **环境变量（可选）**：`PLAN_OUTLINE_SINGLE_CALL_MAX`、`OUTLINE_SKELETON_BATCHES`、`OUTLINE_SKELETON_RECENT_CONTEXT`、`OUTLINE_WINDOW_SIZE`、`OUTLINE_REPAIR_BACK_CHAPTERS`。
- **返回**：`{outline_structure, outline, outline_generated_until}`（outline 为可读字符串）；可能含 **`canon_overrides`**。

### 7.2.1 plan_outline_extend（滑动窗口扩窗）

- **触发时机**：`POST /projects/{project_id}/chapters/next` 前，若 `current_chapter_index + OUTLINE_TRIGGER_MARGIN > outline_generated_until`。
- **输入**：`start_chapter`、`extend_count`、`recent_fact_pack`（最近摘要/最近大纲点/人物快照/硬约束卡）及可选 `kb_context`。
- **行为**：
  - 生成新增区间章节，并输出连贯字段：`depends_on/carry_forward/new_threads/resolved_threads`；
  - 允许回修边界前 `OUTLINE_REPAIR_BACK_CHAPTERS` 章（仅改 points 与线索字段，不改 title）；
  - 对 `last_written_chapter_index` 及之前章节自动禁回修，避免“大纲更新但正文未同步”；
  - 合并前执行结构校验（索引、depends_on 不指向未来、字段完整）；失败触发重试，不落盘。
  - 合并后仅对新增/回修索引区间回写 RAG。
- **返回**：`{outline_structure, outline, outline_generated_until, outline_extended_indices}`。

### 7.3 write_chapter

- **输入**：`project_id`、`current_chapter_index`、`outline_structure`、`chapter_word_target`；可选 `rag_retriever`、`graph_store`；可选 **`global_kb_retriever` + `planner_llm`**（服务端注入，用于混合检索与受限补查）。
- **行为**：
  1. 从 outline_structure 按序号取当前章的 title、points；
  2. 调用 `rag_retriever.retrieve_for_chapter(...)` 得到前文摘要与当前章大纲片段；
  3. 从 CharacterGraphStore 读取“上一章快照”，并按 `CHARACTER_GRAPH_RECENT_CHAPTERS` 做关系滑窗过滤，拼成简短「人物与关系」摘要；
  4. 若 `kb_enabled` 且已选知识库：调用 `build_kb_context_for_writing` 得到资产摘要 + 证据文本；prompt 明确 **二创改动（`canon_overrides`）> 本项目已写事实 > 原著知识**；
  5. 拼 prompt（章节标题、要点、前文摘要、大纲片段、人物摘要、知识块），要求输出 Markdown；
  6. 调用 writer_llm 生成正文，写入 ChapterStore，构造 ChapterMeta 并更新 state 的 chapters。
- **返回**：`current_chapter_draft`、`chapters`、以及 retrieved/character 等中间结果（便于调试）；可能含 **`kb_*` 与 `consistency_report`**。

### 7.4 refine_chapter

- **输入**：当前章内容（从 state 或 chapter_store 读）；state 中可选 **`canon_overrides`** 与知识库节选（`kb_assets_text` / `kb_evidence_text`）。
- **行为**：在文风润色基础上做**轻量一致性守护**：若与原著证据冲突但以二创为准，不回退剧情；保持 Markdown，写回 chapter_store。
- **返回**：`current_chapter_final` 等。

### 7.5 post_chapter

- **输入**：当前章正文、project_id、current_chapter_index。
- **行为**：
  1. 用 LLM 生成 200～500 字摘要，调用 `rag_indexer.add_chapter_summary(...)`；
  2. 用 LLM 从正文抽取人物与关系（JSON），为新关系标注 `first_chapter=current_idx`；
  3. 基于“上一章快照”合并并写入当前章快照：`CharacterGraphStore.merge(..., chapter_index=current_idx, base_graph=...)`；
  4. 可选：从正文再抽取**新增二创设定偏移**，合并进 **`canon_overrides`**，供后续章节检索优先使用。
- **返回**：`last_chapter_summary`、`character_graph`；可能更新 **`canon_overrides`**。

### 7.6 rewrite_with_feedback

- **输入**：`user_feedback`、当前章全文（state 或 store）。
- **行为**：按反馈重写当前章，保持 Markdown，写回 chapter_store。
- **返回**：`last_rewrite_draft`、更新后的 chapters 等。

### 7.7 update_outline_from_feedback

- **输入**：`user_feedback`、`last_rewrite_draft`、`outline_structure`、`current_chapter_index`。
- **行为**：用 LLM 只更新当前章（及可选后续 1～2 章）的要点，写回 state 的 outline_structure。
- **补充**：若注入了 `rag_indexer`，会把受影响章节重新 upsert 到 RAG（保持写章检索与最新大纲一致）。
- **返回**：`{outline_structure: ...}`。

### 7.8 插图三件套（可选）

- **identify_illustration_points**：当 `enable_chapter_illustrations=True` 时，用 LLM 输出 **一处** 最佳插入位置（`illustration_point`：位置说明、锚点原文、alt）与 **文生图详细描述**（`illustration_prompt`）；并写入兼容列表 `illustration_points`（单元素）。**开关为 False 时直接返回空，不调用 LLM**。提示词要求优先**情节高潮/转折/强情绪**等与故事强相关的段落，并要求 `illustration_prompt` 与锚点场景一致；若 state 含 `outline_structure`，会附带**本章大纲要点**供选题（不写入锚点）。解析后对 `anchor_text` 做**向正文收敛**（截成可命中前缀），减轻模型抄错尾部的问题。LLM 失败时的 fallback 优先从**非标题段落的中后段**取锚点，减少总落在篇首的情况。
- **fetch_or_generate_images**（别名 **generate_chapter_illustration_node**）：读取 `illustration_prompt`（或列表首项的 `image_query`），调用 `illustration/openai_generate.py` 写入 `project_id/chapters/{章号}/images/`；成功则填充 `illustration_assets`（单张）与 `illustration_asset`。**OpenAI 调用失败则返回空列表，不插入图、不用占位图**。
- **insert_illustrations_into_chapter**：在锚点对应片段**结束之后**插入 **HTML `<img class="chapter-illustration" ... title="生图提示词">`**（`src` 为相对 `PROJECTS_ROOT` 的路径），提示词持久化在 `title` 供前端悬停展示；若无可用 `image_path` 则不改正文。锚点定位支持**精确匹配**与多种**宽松匹配**（英文多空格、换行差异、汉字间多空格而正文连续等），避免锚点略不一致时整图被追加到章节末尾。仍兼容旧版 `![alt](path)` 纯 Markdown 插图。

**server.py**：`generate_chapter_for_current_index` 在 **refine_chapter 之后** 调用 `run_illustration_pipeline_after_refine`（上述三节点）。**反馈重写**接口顺序为：rewrite →（可选）update_outline → **refine_chapter** → 同上插图流水线 → post_chapter。请求体若带 `enable_chapter_illustrations` 则覆盖本次 state（与续写/重生成一致）。

静态访问：后端挂载 **`/project-data`** → `PROJECTS_ROOT`，前端将正文中的 `project_id/chapters/...` 图片路径解析为同源 **`/project-data/{相对路径}`**。

## 八、工作流组装（graph/workflow.py）

- **StateGraph(NovelProjectState)**，节点即上述各 node 的同步包装（内部 `asyncio.run(节点(state, ...))`）。
- **build_stage2_workflow**：generate_plot_ideas → plan_outline → write_chapter → refine_chapter → END（无 RAG、无 post、无反馈）。
- **build_stage3_workflow**：stage2 + refine 后，若 `enable_chapter_illustrations` 则 **identify → fetch_or_generate_images → insert** → `post_chapter_update`，否则 refine 后直接 post。
- **build_stage4_workflow**：write → refine → 若有 `user_feedback` 则 **rewrite_with_feedback → refine_after_rewrite**（第二次润色）；再按需 **update_outline_from_feedback**；最后统一经 **插图链（同上）或直跳** → `post_chapter_update` → END。
- **build_stage7_workflow**：与 stage4 相同的反馈/二次润色/大纲更新结构，插图链与 stage3/4 一致（`enable_chapter_illustrations=False` 时 identify 空操作后 insert 无资源即跳过）。

**说明**：`workflow.py` 内的 `write_chapter` / `plan_outline` 包装**未传入** `global_kb_retriever` 等知识库依赖；**同人知识库能力以 `server.py` 手写流水线为准**。命令行/编译图路径若需 KB，需自行注入与 HTTP 服务相同的参数。

当前 **server.py 并未直接使用这些 build_* 的编译图**，而是按接口「手写」调用顺序：续写/重生成 = write → refine →（可选）插图三节点 → post；**重写** = rewrite →（可选）update_outline → **refine** →（可选）插图三节点 → post，再 save_state。

补充说明：

- 所有写操作都会在最后 **save_state(project_id, state)**，保证刷新后状态一致。
- 关键写作动作（生成概要、生成大纲、续写下一章、反馈重写）会写项目事件日志：`projects/{project_id}/event_logs.ndjson`，事件字段含 `event_name`、`event_content`（截断后的简短文本）、`ts`、`status`、可选 `chapter_index`。
- `rewrite` / `regenerate` 均有「仅最新章可操作」约束（后端 `ensure_latest_chapter_only`）。
- 项目显示名规则：前端使用 `nickname || project_id`；`project_id` 仍是唯一稳定主键，删除/读写等 API 仍使用 `project_id`。
- 多个长耗时接口支持可选查询参数 **`stream`**（布尔，FastAPI）：`stream=1`、`stream=true` 等均可：返回 `application/x-ndjson`（NDJSON）。
  - 流中每行形如 `{"type":"progress","stage":"...","message":"..."}` 用于展示当前步骤；
  - 最后一行为 `{"type":"result","body":{...}}`，与普通 JSON 返回结构一致。
  - 若 LLM 或解析失败，最后一行可为 `{"type":"error","detail":"...","status":?}`（由 `ndjson_with_progress` 的 worker 捕获异常后写入）。
  - `ndjson_with_progress` 内进度通道使用**无界** `asyncio.Queue`：润色/重写等会按 token 高频 `emit`，若使用有界队列且 HTTP 写出因 TCP 反压阻塞，会出现 worker 与消费端互相等待的死锁，前端表现为长时间无 result/error。
  - **反馈重写** `?stream=1`：token 级增量**仅**来自 **`refine_chapter_stream`**（润色「按反馈重写」后的本章）；`rewrite_with_feedback` 阶段不推送正文 token 流（仅有阶段型进度 `rewrite`），避免前端流式预览停在重写稿、与最终润色稿不一致。
  - 生成大纲在章节数超过 `PLAN_OUTLINE_SINGLE_CALL_MAX` 时，`plan_outline` 阶段会出现「骨架」与按批「扩写要点」等多条 progress（由 `plan_outline_node` 的 `on_progress` 回调驱动）。
- `GET /projects` 前会执行空项目清理：同时满足「**无大纲、无章节**（含无章节元数据且无 `chapters/` 下 .md）」「**无任何创作活动痕迹**（`instruction`、`plot_ideas`、`selected_plot_summary` 均为空）」且自创建起超过 **10 分钟**（`created_at` 或状态文件/项目目录 mtime 推算）的项目会被删除，并同步删除 `PROJECTS_ROOT` 下该项目目录与 `VECTOR_STORE_DIR` 下对应向量目录。
- `DELETE /projects/{project_id}` 走显式硬删除：删除 `checkpoints/api_state/{project_id}.json`、`projects/{project_id}`、`vector_store/{project_id}`，并对 `_video_background_tasks` 中该项目任务先发取消信号再 `task.cancel()`，减少删除后后台继续写盘的风险。

  - `state` 维护 currentProjectId、plotIdeas、selectedIdea、selectedChapterIndex、chapterMetas、写作中状态锁、**currentProjectKbBindLocked** / **currentProjectKbIds**、知识库列表等；

  - 续写下一章、重新生成本章、反馈重写请求体均携带 **`enable_chapter_illustrations`**（由复选框决定，默认不勾选）；
  - 知识库：`loadKnowledgeBases`、创建后 `POST /knowledge-bases`、上传 `multipart` 至 `documents`、轮询 `jobs`；无当前项目或**已生成大纲**时禁用「绑定当前项目」勾选（与后端 PATCH 规则一致）；新建项目时若已勾选知识集，随 `POST /projects` 提交 **selected_kb_ids**；
  - 项目列表支持右键自定义浮层菜单：`重命名`（调用 `PATCH /projects/{project_id}` 更新 `nickname`）与 `删除项目`（调用 `DELETE /projects/{project_id}` 执行硬删除）；
  - 项目列表加载、打开项目、生成概要（复用当前项目）、自定义概要、生成大纲、续写下一章、重生成、回滚、反馈重写、人物图谱弹窗等，均通过 API 驱动；
  - 交互约束：当选中非最新章时，自动隐藏“续写/重生成/重写”，仅显示“回滚到本章”。
- **styles.css**：布局与样式。

前端与后端通过 REST 通信；长耗时操作（生成大纲、续写、重写）支持 `?stream=1` 的 NDJSON 进度流，前端会持续更新 `#global-status`，直到最终结果返回并渲染到页面。**反馈重写**开启流式时，正文区仅跟随 **`refine_chapter_stream`** 更新（与上条 API 约定一致）。

## 十一、数据流小结

1. **新建小说**：（可选）创建/上传全局知识库并勾选绑定 → 创建项目（可带 **selected_kb_ids**）→ POST plot-ideas（instruction）→ 选或填 selected_plot_summary → POST outline（服务端注入 **kb_context**）→ 得到 outline_structure，并可能写入 **canon_overrides**。
2. **写第一章**：POST chapters/next → current_chapter_index=0（先过 `ensure_outline_window`，首章通常无需扩窗）→ write_chapter（RAG 暂无前文；人物图谱读取上一章快照为空）→ refine → 可选插图 → post_chapter（摘要入 RAG、图谱写入第 0 章快照）→ save_state。
3. **写第 N 章**：POST chapters/next → current_chapter_index=N；若接近边界则自动执行 `plan_outline_extend`（基于 recent_fact_pack 扩窗并回写新增索引 RAG）→ write_chapter 检索前 k 章摘要 + 当前章大纲片段 + 最近关系滑窗 → refine → 可选插图 → post_chapter → save_state。
4. **反馈重写**：POST chapters/{index}/rewrite（仅最新章）→ rewrite_with_feedback → 若 update_outline 则 update_outline_from_feedback（并回写受影响章的大纲 chunk）→ post_chapter → save_state。
5. **重新生成**：POST chapters/{index}/regenerate（仅最新章）→ 复用“写章流水线”覆盖本章 → save_state。
6. **章节回滚**：DELETE chapters/{index}/tail → 删除 index 之后章节文件 → 删除对应 RAG 摘要与人物图谱快照尾部 → 更新 state 当前章节指针。

## 十二、短篇模式（run.py）

- 使用独立的 **StroyState**（instruction、outline、draft、final_stroy）和一条简单链：planner → writer → refiner → END。
- 不涉及项目、章节存储、RAG、人物图谱，适合快速体验「意图 → 大纲 → 初稿 → 润色」的流水线，与长篇分章 Agent 共用 config 中的模型配置即可。
